```
# Clarifying Questions - Module 3: Hidden Critical Ambiguities
## Result Summary

**Completed**: Yes  
**Format**: Four exercises with detailed requirements hiding dangerous production-level ambiguities

---

### Exercise 1: Automated Daily Database Backup
Participant asked: full DB vs specific tables, encryption key management, large database handling, partial failure and resume, S3 failure and redundancy, parallel backup optimization.

**Strengths**: Strong infrastructure and redundancy thinking — S3 failure, parallel backups, partial failure resume. Encryption key management is an expert-level question that most people skip.  
**Gaps**: Missed DST/timezone handling for "2 AM PST" (runs twice or skips during transition), overlapping backup runs, backup integrity verification and restore testing, silent failure monitoring, and access control on backup retrieval.

---

### Exercise 2: Stripe Payment Processing Integration
Participant asked: retry exhaustion handling, payment success + DB update failure, email delivery failure, plan pricing specifications, bulk transaction DB write handling, cross-region payment consistency.

**Strengths**: Correctly identified the payment success + DB failure scenario — the single most dangerous distributed systems failure in payment integrations. Also caught retry exhaustion policy and cross-region consistency.  
**Gaps**: Missed webhook idempotency (Stripe webhooks can arrive delayed, duplicated, or not at all — processing the same event twice can double-charge), race condition when user cancels during active retry, refund impact on subscription lifecycle, PCI compliance requirements.

---

### Exercise 3: Real-Time Analytics Dashboard
Participant asked: existing metrics infrastructure (Grafana), concurrent WebSocket connection limits, alert frequency and suppression, alert delivery channels, WebSocket failure and data recovery, data loss retry vs skip.

**Strengths**: Alert suppression/fatigue was the standout — recognising that cascading failures can trigger every threshold simultaneously and spam alerts into being ignored. Also correctly identified WebSocket connection limits as a scalability ceiling.  
**Gaps**: Missed performance impact of metric collection on the production system being monitored (self-defeating architecture), real-time vs historical data consistency conflict, and graceful degradation if dashboard itself fails.

---

### Exercise 4: Multi-Tenant Data Migration
Participant asked: full vs partial rollback, migration performance impact on live system, data changes during migration, backup server redundancy.

**Strengths**: Partial rollback danger (mixed state) and data-during-migration (incremental vs big-bang problem) were both correct and important.  
**Gaps**: Missed tenant data isolation breakage risk (mixing customer data is a legal/compliance crisis), external system dependencies on legacy IDs/formats that silently break post-migration, rollback destroying new data created after migration started, compliance/data residency requirements in multi-tenant architecture.

---

### Final Challenge
Participant chose the migration exercise and wrote three targeted questions:
1. Is there a way to detect multi-tenancy breakage and trigger rollback automatically?
2. What is the legacy system's format and what external dependencies does it have?
3. How do we protect newly created data if a rollback is triggered?

All three questions target real catastrophic failure modes: silent tenant data mixing, external dependency breakage, and the rollback paradox. Question 3 (rollback destroying new data) was the sharpest — it exposes the fundamental tension in the requirement's "rollback capability" promise.

---

### Overall Pattern (Module 3)
Consistently strong: infrastructure redundancy, partial/mid-operation failure handling, scalability limits, alert management.  
Consistently missing: boundary conditions — time boundaries (DST, overlapping runs), system boundaries (webhooks, external dependencies), tenant boundaries (data isolation, compliance).  

Key insight reinforced: the most dangerous hidden risks live at boundaries between systems, between time windows, and between tenants. Detailed-looking requirements often hide these boundary conditions completely.
```

```
# Stakeholder Conversations - Module 3: The Contradicting Stakeholders
## Result Summary

**Completed**: Yes  
**Scenario**: Two stakeholders with directly opposing requirements for an order management system rebuild. Dana (Ops) wants full automation and no manual steps. Raj (Sales) wants mandatory human review on every order after a $200K pricing incident.

---

### Learner Approach
Asked strong diagnostic questions with both stakeholders — uncovered Dana's verification process details and Raj's minimum requirements (pricing tier accuracy on contracted clients, orders above $10K). Got to the tiered verification synthesis by asking Raj what he could not live without, rather than proposing a solution upfront.

---

### Debrief Results

**1. Spotting the contradiction — Partial**
Recognised the tension between the two requirements but did not name it explicitly to either stakeholder. Explicitly surfacing the conflict ("Dana needs auto-push, you need mandatory review — these are directly in tension") prevents each side from assuming you agreed with them.

**2. Gathering the "why" — Strong**
Asked the right diagnostic questions with both stakeholders. With Dana: what does verification actually catch, how do duplicates happen, what's the cost of a bad order. With Raj: what specific steps can't go wrong. Raj's answer (pricing tier accuracy on contracted clients and high-value orders) revealed the real constraint and made the synthesis possible.

**3. Finding the middle ground — Strong**
Reached the tiered verification model by asking Raj what his minimum requirements were rather than imposing a solution. He identified the threshold himself: orders above $10K and contracted clients need review; everything else can auto-push. That satisfies both stakeholders' underlying needs.

**4. Avoiding picking sides — One slip**
Told Raj "I already talked to Dana, she's all on the speed train" — this created mild stakeholder tension. Raj immediately pushed back. The tech lead synthesises; they do not relay one stakeholder's position to another as a negotiating lever.

**5. System design implications — Not raised**
The tiered model requires: order value thresholds, contract flags per client, routing logic for auto-push vs review queue, and an audit trail. These were not articulated back to Raj to confirm the spec.

---

### Learner Takeaway
"I had a preconceived notion about automating everything — I went into the Raj conversation already sold on Dana's solution and tried to push it on him rather than genuinely understanding his position first. I need to understand the 'why' behind each stakeholder before advocating for anything."
```
