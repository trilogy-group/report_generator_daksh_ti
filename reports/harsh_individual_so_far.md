## 4. Customer Pain vs Developmental Details

The understanding-problems track shows strong customer-pain framing. Deep-requirements module 3 (billing case 1):

> "(1) User communication about billing changes, (2) refund policy for 'unused time,' (3) edge cases by subscription type (annual, trial, mid-promo), (4) immediacy vs next-cycle application."

The communication requirement—customers need to understand what they're being charged—is a customer-pain insight, not a technical implementation detail. This is above average in the cohort.

Deep-requirements module 1 combined two scenario branches:

> "Chose B (price/availability changes → alerts, availability notifications) and C (purchase triggers → reminders, idle-cart, sale notifications). Combined both branches."

That combination—price-drop alerts AND purchase triggers as dual customer motivations for the save-for-later feature—shows customer-workflow thinking, not feature thinking. Strong.

The gap: in the solving track (all 6 decomposition modules), customer-pain framing is largely absent. The architectures are correctly structured but don't specify what the customer experiences when stations fail.
